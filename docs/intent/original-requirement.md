# wljob 项目意图与测试描述

本文档用于作为 `wljob` 初始化、Nexus 调研和 Verix 审计时的项目意图输入。`wlzuu` 是早期由 skhub 生成的需求脚手架和历史来源；当前测试应以 `wljob` 的具体目标、范围和安全边界为准。

## 背景关系

- `wlzuu`：早期需求探索项目，记录了“构建中文互联网求职辅助 agent”的原始想法和边界要求。
- `wljob`：承接 `wlzuu` 意图的受控实现目标，是当前更具体的 workflow/tool 设计依据，也是 Nexus/Verix 端到端测试的目标项目名。

## 项目目标

`wljob` 目标是构建一个实用、可审计、面向中文互联网求职和网申流程的本地 workflow/tool。Codex 或宿主模型负责意图理解、路线选择、计划生成和风险审查；本地代码负责受控执行、状态记录、artifact 输出、岗位信息整理和审批门禁。

核心场景包括：

- 根据自然语言理解用户求职意图和默认画像。
- 在公司官网招聘入口、官方 ATS 页面、Boss 直聘类平台、用户提供的岗位链接和其他可扩展来源之间选择访问路线。
- 整理岗位信息，先输出可读中文摘要，再输出结构化 JSON 或表格。
- 在任何敏感动作前生成 proposal/confirmation card。
- 对外部副作用进行显式审批和审计记录。

## 默认用户画像

- 求职方向：后端工程、工程型算法、数据分析、AI 大模型工程化应用。
- 关注场景：智能体平台、开发者 AI、AI Infra、数据与评测、智能驾驶相关岗位。
- 输出偏好：先给简洁可读中文总结，再给结构化 JSON、表格或岗位卡片。

## 必备能力

- 意图 intake：解析用户求职目标、约束、公司/岗位/链接输入。
- 路线选择：判断应使用公司官网、官方 ATS、招聘平台、用户链接或本地样例数据。
- 搜索/来源规划：生成受控检索计划，不伪造 URL 或岗位来源。
- 岗位卡片标准化：抽取公司、岗位、地点、薪资、技能、链接、申请状态、风险提示等字段。
- 岗位评分与排序：结合默认画像和用户偏好对岗位做解释性评分。
- proposal 生成：对下一步操作给出可读建议和审批卡。
- 状态与审计：输出 `state.json`、`job_cards.json`、`proposals.json`、`approval_required.json`、`audit.json`、`interaction.json` 或 `blocked.json` 等 artifact。
- 可扩展 adapter：为公司官网、ATS 页面、招聘平台和本地数据源预留扩展点。

## 硬安全边界

- 不读取 cookie、token、浏览器 profile、SSH key、`.env`、密码文件或任何凭据。
- 不绕过 CAPTCHA、403、WAF、登录墙、平台风控或反爬系统。
- 不自动输入账号密码。
- 不自动提交网申。
- 不自动发送消息或联系招聘方。
- 登录和身份校验只能由用户手动完成。
- 简历上传、附件上传、最终投递点击、发送沟通内容、外部写入、发布、安装依赖、真实浏览器访问、外部 LLM 调用等动作都必须单独审批。
- 如果遇到登录、验证码、风控、异常跳转、敏感页面或凭据请求，必须停止并写入 blocked/approval artifact。

## MVP 范围

当前最小可用范围应优先验证本地、可审计的 workflow 行为：

- 解析自然语言求职请求。
- 生成受控搜索/来源计划。
- 将已批准的本地或只读来源数据整理成岗位卡片。
- 生成下一步 proposal 和审批卡。
- 记录状态、审计和交互 artifact。
- 在没有真实 provider、没有批准来源、请求不安全或触发风控时输出 blocked，而不是静默使用 mock 或编造结果。

## 非目标

- 不做自动投递系统。
- 不做自动私信招聘方系统。
- 不做登录自动化或密码输入。
- 不做验证码、403、WAF 或风控绕过。
- 不在缺少真实 provider/API key 时编造带 URL 的线上候选。
- 不把 `wljob` 当成 `wlzuu` 脚手架的简单复制；`wljob` 应服务于验证中文互联网求职/网申 workflow kernel 的工程能力。
